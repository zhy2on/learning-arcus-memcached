# arcus-memcached 엔진 인터페이스 구조

Arcus memcached 엔진 코드를 보다 보면 `mc_engine.v0`, `mc_engine.v1`, `ENGINE_HANDLE`, `ENGINE_HANDLE_V1` 같은 표현이 계속 나온다. 이 문서는 그 구조를 이해하기 위한 메모다.

---

## 한 줄 요약

`v1`은 **실제 함수 포인터 집합을 담은 인터페이스 구조체**이고, `v0`는 **그 구조체 맨 앞에 놓인 최소 공통 헤더 타입**이다.

Arcus가 `v0`/`v1`을 나눈 핵심 이유는, 엔진을 **공통 핸들 타입으로 받고**, 그 헤더를 기준으로 **안전하게 `ENGINE_HANDLE_V1`로 캐스팅**하기 위해서다.

---

## 왜 `v0`, `v1`로 나누나?

현재 Arcus 코드 기준으로 보면, 이 구조의 핵심은 **공통 헤더를 맨 앞에 두어 binary compatibility를 유지하는 것**에 있다.

`doc/engine-interface.xml`에는 `ENGINE_HANDLE_V1`의 첫 멤버 `interface`에 대해 이렇게 적혀 있다.

- binary compatibility를 보존하려면 반드시 구조체의 첫 멤버여야 한다
- 이 값을 1로 설정해야 한다

즉 의도는:

1. 서버는 엔진을 일단 `ENGINE_HANDLE*`라는 **공통 타입**으로 받는다.
2. 그 공통 헤더의 `interface` 값을 본다.
3. 현재 구현이 기대하는 버전이면 `ENGINE_HANDLE_V1*`로 캐스팅해서 사용한다.

실제 `engine_loader.c`도 정확히 이렇게 동작한다.

```c
if (engine->interface == 1) {
    engine_v1 = (ENGINE_HANDLE_V1*)engine;
    ...
}
```

즉 `v0`/`v1` 분리의 핵심은:

- 플러그인 엔진을 공통 타입으로 주고받고
- 그 구조체의 시작 부분 레이아웃을 고정해 두고
- 그 위에 더 큰 인터페이스 구조체를 얹어 캐스팅 가능하게 만드는 것

이라고 보는 게 현재 코드 기준으로 가장 정확하다.

> [!NOTE]
> 여기서 문서와 현재 구현을 구분해서 보는 게 중요하다.
>
> 문서로 확실히 확인되는 것은:
> - `ENGINE_HANDLE`이 공통 헤더라는 점
> - `create_instance()`가 서버가 지원하는 highest interface level을 인자로 받는다는 점
> - `ENGINE_HANDLE_V1`의 첫 멤버가 공통 헤더여야 하고 이유가 binary compatibility라는 점
>
> 반면 현재 Arcus 코드가 실제로 하는 일은:
> - `interface == 1`인지 확인한 뒤
> - `ENGINE_HANDLE_V1*`로 캐스팅해서
> - 사실상 `v1`만 사용하는 것
>
> 즉 "버전 확장을 고려한 인터페이스 모양"은 있지만, "현재 구현이 v2/v3를 런타임에 동적으로 선택한다"까지는 아니다.

---

## 그런데 왜 함수 호출할 때는 `v0`를 넘기나?

이건 **버전 확인 때문이 아니라 C에 `this`가 없기 때문**이다.

C++ 멤버 함수처럼:

```cpp
engine->get(key, nkey);
```

가 되는 게 아니라, C에서는 함수가 자기 인스턴스를 스스로 알 수 없으니 첫 번째 인자로 명시적으로 넘겨줘야 한다.

그래서 호출은 이런 형태가 된다.

```c
mc_engine.v1->get(mc_engine.v0, key, nkey);
```

여기서 의미는:

- `mc_engine.v1`
  어떤 함수를 호출할지 찾기 위한 함수 포인터 테이블
- `mc_engine.v0`
  그 함수를 어느 엔진 인스턴스에 대해 호출할지 알려주는 포인터

즉 `mc_engine.v0`는 함수 호출 시 사실상 C에서의 `this` 대체처럼 동작한다. 다만 Arcus에서 `v0`를 따로 둔 근본 이유는 "this를 흉내 내기 위해서"보다, **공통 헤더 타입을 맨 앞에 둔 인터페이스 구조를 만들기 위해서**라고 보는 편이 정확하다.

---

## `ENGINE_HANDLE`로는 뭘 할 수 있나?

`ENGINE_HANDLE` 자체는 아주 작은 공통 핸들이다. 실제 정의를 보면:

```c
typedef struct engine_interface {
    uint64_t interface;
} ENGINE_HANDLE;
```

즉 `ENGINE_HANDLE`만 놓고 보면 할 수 있는 일은 많지 않다. 사실상 담고 있는 건:

- 이 포인터가 "엔진 인터페이스 핸들"이라는 공통 의미
- 인터페이스 버전 번호

정도다.

그래서 `ENGINE_HANDLE`의 역할을 정확히 말하면:

- 서버가 엔진 인스턴스를 공통 타입으로 다룰 수 있게 해 주고
- 현재 로더가 기대하는 인터페이스 버전인지 확인하게 해 주는 최소 공통 핸들

이다.

여기서 중요한 점은:

- "`default` 엔진인지 `demo` 엔진인지"를 직접 알려주는 건 아님
- "이 포인터는 엔진 핸들이고, 이 버전의 인터페이스로 다뤄야 한다"를 알려주는 것에 가깝다

즉 `ENGINE_HANDLE`은 엔진의 상세 정체성을 설명하는 객체라기보다, **서버와 엔진이 공통 규약으로 주고받는 최소 단위 핸들**이라고 보면 된다.

---

## 왜 `&mc_engine`이 아니라 `mc_engine.v0`인가?

질문이 자연스럽게 드는 지점이 있다.

"C에 `this`가 없으면 그냥 자기 자신의 주소를 넘기면 되는 거 아닌가? 예를 들어 `&mc_engine` 같은 식으로."

그런데 중요한 건 엔진 함수가 원하는 타입이 그냥 "아무 구조체의 주소"가 아니라는 점이다. 예를 들어 `get` 함수 포인터 시그니처는:

```c
ENGINE_ERROR_CODE (*get)(ENGINE_HANDLE* handle, const void* cookie,
                         item** item, const void* key, const int nkey,
                         uint16_t vbucket);
```

처럼 **첫 번째 인자로 `ENGINE_HANDLE*`를 요구**한다.

즉 함수는:

- "엔진 인터페이스 핸들 포인터"

를 원하지,

- "바깥 래퍼 구조체 전체 주소"

를 원하는 게 아니다.

그리고 default 엔진 구현체를 보면 실제 엔진 인스턴스는:

```c
struct default_engine {
   ENGINE_HANDLE_V1 engine;
   ...
};
```

처럼 생겼다. 즉 엔진 객체의 맨 앞에 `ENGINE_HANDLE_V1 engine`이 들어 있다.

그래서 함수 호출에는:

- `v1`로 함수 포인터를 찾고
- `v0`로 함수가 기대하는 공통 핸들 타입을 넘기는 것

이 자연스럽다.

한마디로 말하면:

- `&mc_engine`은 문맥에 따라 아예 타입이 다를 수 있고
- 함수는 표준 인터페이스 규약상 `ENGINE_HANDLE*`를 원하므로
- 그에 맞는 포인터인 `mc_engine.v0`를 넘기는 것이다.

---

## `v0`가 두 역할을 같이 할 수 있는 이유

같은 엔진 객체의 시작 주소를 서로 다른 타입으로 해석하기 때문이다.

즉:

- `v0`로 보면 공통 헤더
- `v1`로 보면 공통 헤더 뒤에 함수 포인터 테이블까지 포함한 더 큰 인터페이스

처럼 다룰 수 있다.

현재 Arcus 코드에서는 **`interface == 1`인지 확인한 뒤 `ENGINE_HANDLE_V1*`로 캐스팅**하는 형태로 쓰인다.

즉:

- `v0`는 공통 헤더/캐스팅 기준점
- `v1`은 실제 함수 포인터 구조체
- 같은 시작 주소를 공유하므로 `ENGINE_HANDLE*`에서 `ENGINE_HANDLE_V1*`로 해석할 수 있음

이 포인트가 중요하다.

---

## 비유로 보면

- `v0` = 공통 신분증 + 인스턴스 본체를 가리키는 손잡이
- `v1` = 그 엔진이 제공하는 기능 목록표

코어는 먼저 신분증을 보고:

- "이 엔진이 내가 기대하는 인터페이스 형식인가?"

를 확인한 다음, 그 기능 목록표에서 필요한 함수를 찾아 호출한다.

호출할 때는 다시 "이 함수가 어느 엔진 인스턴스에 대해 실행돼야 하는가"를 알려주기 위해 `v0`를 넘긴다.

---

## GET 흐름에서 보면

`engine-get.md`에서 보는:

```c
mc_engine.v1->get(...)
```

라는 표현은 단순히 "v1 인터페이스의 get 함수를 호출한다"는 뜻이다.

좀 더 풀어보면:

1. `v1`에서 `get` 함수 포인터를 꺼내고
2. 그 함수의 첫 번째 인자로 `v0`를 넘겨
3. 이 호출이 어느 엔진 인스턴스에 대한 것인지 알려주는 구조다.

그래서 GET 흐름을 볼 때도:

- `v1`은 함수 찾기
- `v0`는 엔진 인스턴스 넘기기

로 이해하면 읽기 쉬워진다.

---

## 말로 설명하는 정리본

"Arcus에서 `v0`와 `v1`을 나눈 핵심 이유는 공통 헤더를 구조체 맨 앞에 둬서 엔진을 `ENGINE_HANDLE*`라는 공통 타입으로 주고받고, 그걸 안전하게 `ENGINE_HANDLE_V1*`로 캐스팅하기 위해서다. `v0`는 인터페이스 버전 번호를 담은 최소 공통 헤더이고, `v1`은 그 뒤에 실제 함수 포인터들을 붙인 구조체다. 현재 Arcus 구현은 사실상 v1만 쓰지만, 로더는 먼저 `ENGINE_HANDLE`의 `interface` 값을 보고 그다음 `ENGINE_HANDLE_V1`로 캐스팅해서 함수 포인터를 사용한다. 함수 호출 시 `mc_engine.v0`를 넘기는 건 그 공통 핸들 포인터를 첫 번째 인자로 주는 것이고, 결과적으로 C에서의 `this`처럼 동작한다."

---

## 워크북

#### 1. `v0`와 `v1`을 굳이 나눈 가장 큰 이유는 무엇인가?

<details>
<summary>답안 보기</summary>

> [!NOTE]
> 공통 헤더를 구조체 맨 앞에 둬서 binary compatibility를 유지하고, 서버가 엔진을 `ENGINE_HANDLE*`라는 공통 타입으로 받은 뒤 `ENGINE_HANDLE_V1*`로 안전하게 캐스팅할 수 있게 하기 위해서다.

</details>

#### 2. `v0`의 두 가지 역할은 무엇인가?

<details>
<summary>답안 보기</summary>

> [!NOTE]
> 공통 헤더 역할과, 함수 호출 시 첫 번째 인자로 넘기는 핸들 역할이다. 현재 코드에서 결과적으로 `this` 대체처럼 동작한다.

</details>

#### 3. `mc_engine.v1->get(mc_engine.v0, ...)`에서 `v1`과 `v0`는 각각 무엇을 의미하는가?

<details>
<summary>답안 보기</summary>

> [!NOTE]
> `v1`은 호출할 함수 포인터를 찾는 인터페이스 테이블이고, `v0`는 그 함수를 어느 엔진 인스턴스에 대해 호출할지 알려주는 포인터다.

</details>

#### 4. 함수 호출 때 `v0`를 넘기는 이유는 버전 확인 때문인가?

<details>
<summary>답안 보기</summary>

> [!NOTE]
> 아니다. 함수 호출 시 `v0`를 넘기는 핵심 이유는 C에 `this`가 없어서 호출 대상 엔진 인스턴스를 첫 번째 인자로 명시해야 하기 때문이다.

</details>
